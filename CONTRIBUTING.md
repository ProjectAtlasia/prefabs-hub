# Contributing

## Branches

- **`develop`** — integration branch where features accumulate.
- **`master`** — released code. Protected: changes land only via pull request and the `build` check
  must pass. Pushing to `master` triggers a release (see below).

## Feature flow

1. Branch off `develop`: `feat/<thing>`, `fix/<thing>`, `chore/<thing>`, …
2. Open a PR **into `develop`**. The `validate` workflow runs (build, format, security scanners).
3. **Squash-merge** with a Conventional Commit title — one feature = one commit = one changelog entry.

## Conventional Commits (these drive the version)

Format: `type(scope): subject` — one line, English, imperative, lowercase after `:`, no body unless
indispensable, **no `Co-Authored-By` trailer**.

| Type | Release effect |
|------|----------------|
| `fix:` | patch (`0.0.x`) |
| `feat:` | minor (`0.x.0`) |
| `feat!:` or a `BREAKING CHANGE:` footer | major |
| `chore` `ci` `docs` `refactor` `perf` `test` `build` `style` | no release |

To force a specific version (rare, e.g. a re-release), add `Release-As: x.y.z` to the commit body.

## Releasing

1. Open a PR **`develop` → `master`** and merge it with a **merge commit** (not squash — semantic-release
   must see the individual Conventional Commits).
2. The push to `master` runs **semantic-release** (`.github/workflows/release.yml`), which **pauses for
   manual approval** via the `publish` environment.
3. On approval it computes the next version from the commits, creates the tag `v<version>` + GitHub
   release, builds the jar and publishes to CurseForge.

Nothing is committed back to the branches — the version comes from the tag at build time — so `develop`
never drifts and there is **no back-merge**.

## Versioning

The version is derived from the release tag, not stored in source. CI passes
`-PreleaseVersion=<version>`; local builds default to `0.0.0-dev`. Do not commit version literals.

## Building locally

- `./gradlew shadowJar` → fat jar at `build/libs/prefabs-hub-<version>-<buildTime>.jar`.
- On a **fresh clone**, run `./gradlew compileJava` once **before** `shadowJar`: ScaffoldIt reads the
  compiled plugin class to fill the manifest `Main`, so a single clean `shadowJar` would write a wrong
  stub and the jar would not boot. CI does this in two passes automatically.
- Every change must pass `./gradlew compileJava spotlessCheck` (google-java-format, 2 spaces). The
  `.githooks/pre-commit` hook runs `spotlessApply` on staged Java — enable it once per clone with
  `git config core.hooksPath .githooks`.

## Changelog

Release notes are generated by semantic-release into the GitHub release and CurseForge. There is no
maintained `CHANGELOG.md` (committing one would reintroduce branch drift); the file in the repo is a
frozen historical record up to 0.2.2.
